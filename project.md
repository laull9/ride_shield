# 电动车驾驶辅助系统设计

## 1. 项目定位

本项目面向两轮电动车骑行场景，设计一套基于多模态感知与边缘智能推理的驾驶辅助系统。系统部署在瑞芯微 RK3588 车载计算平台上，通过前向环境感知、后向盲区感知、驾驶员状态监测以及姿态和生理信号融合，实现对骑行风险的实时识别、分级预警和有限干预。

与四轮车不同，两轮电动车在急刹、侧倾、视野盲区和驾驶员疲劳方面更敏感，因此系统设计遵循“先预警、后干预、有限制动、安全优先”的原则，避免因激进控制导致二次风险。

本系统强调以下三个比赛亮点：

- 多模态融合：视觉、姿态、毫米波雷达和驾驶员状态联合决策。
- 边缘部署：模型在 RK3588 本地实时运行，不依赖云端。
- 闭环辅助：从感知、识别、决策到语音、振动、屏显和受控减速形成完整闭环。

## 2. 目标场景与核心痛点

系统重点覆盖以下典型风险场景：

1. 前方车辆、行人或障碍物突然接近。
2. 后方来车快速逼近，驾驶员准备变道或转向。
3. 夜间、雨雾或复杂道路下的目标识别、路障识别和车道偏移风险。
4. 驾驶员疲劳、分神、频繁低头或闭眼。
5. 车辆急加速、急转弯、侧倾异常或路面颠簸。
6. 驾驶员心率异常升高或持续异常波动。

对应的系统目标如下：

- 实时感知车前、车后和驾驶员状态。
- 在边缘端完成风险识别与等级评估。
- 通过多级告警降低碰撞和失控风险。
- 在满足安全约束的条件下执行有限自动干预。
- 向车机和手机 App 提供可解释的状态信息与统计结果。

## 3. 系统总体方案

### 3.1 输入与传感器配置

- 前向普通摄像机：采集道路画面，用于目标检测、车道线检测和碰撞风险分析。
- 后向鱼眼摄像机：采集后方广角视野，用于后向来车检测、盲区监测和变道辅助。
- 驾驶员普通摄像机：面向驾驶员，用于疲劳检测、注意力分析和头部姿态识别。
- IMU/陀螺仪：采集横滚角、俯仰角、角速度和振动信息，用于姿态估计与异常骑行判定。
- 毫米波雷达：用于近距离目标辅助测距，并结合面向驾驶员胸部区域的布置方式实现心率检测。

### 3.2 输出与执行器配置

- 语音播报：用于碰撞风险、后方来车、疲劳和心率异常等场景的语音提醒。
- 把手振动：用于驾驶员分神、疲劳、后方盲区来车等需快速提醒场景。
- 车机屏幕：显示风险等级、检测框、车道线、驾驶员状态、姿态信息和系统运行状态。
- 制动系统联动：仅在低速且碰撞概率极高的条件下触发受控减速，不设计高强度突然制动。

## 4. 功能设计

### 4.1 前向环境感知

前向环境感知模块负责识别道路中的关键目标，并估计潜在碰撞风险，主要功能包括：

1. 检测车辆、行人、非机动车、交通锥、施工围挡和交通标志。
2. 检测水坑、坑洼、减速带、落物等道路风险目标。
3. 检测当前车道线和车辆相对车道的偏移趋势。
4. 基于目标框尺寸变化、相对位置和速度趋势估计 TTC（碰撞时间）。
5. 输出“安全、注意、危险、紧急”四级前向风险等级。

### 4.2 后向盲区感知

后向鱼眼摄像机主要用于补足两轮车后向视野不足问题，重点不是做完整自动驾驶感知，而是做实用的后向安全辅助：

1. 检测后方来车、侧后方超车目标和近距离快速接近目标。
2. 识别左后、正后、右后三区域的风险分布。
3. 当驾驶员有转向或变道倾向时，提升后向风险权重。
4. 输出盲区预警和超车预警。

### 4.3 驾驶员监控

驾驶员监控模块结合人脸关键点、眼部状态和头部姿态，识别骑行中的异常状态：

1. 闭眼持续时间检测。
2. 打哈欠和频繁低头识别。
3. 头部偏转和视线偏离估计。
4. 疲劳等级和注意力等级评估。

### 4.4 姿态与生理信号监测

该模块结合 IMU 与毫米波雷达数据，对骑行稳定性和驾驶员生理状态进行辅助判断：

1. 检测急转、急刹、剧烈晃动和异常侧倾。
2. 对道路颠簸和危险操控进行区分，降低误报。
3. 检测驾驶员心率过高、过低或异常波动。
4. 在疲劳与心率异常同时出现时提高系统警报等级。

## 5. 多模态融合与决策机制

### 5.1 融合思路

系统采用“感知结果级融合 + 规则决策 + 可扩展轻量学习模型”的策略，兼顾比赛可实现性与工程可解释性。各模块先独立输出结构化结果，再由决策模块统一评分。

融合输入包括：

- 前向目标类别、目标距离估计、TTC 风险值。
- 后向来车区域、相对速度趋势和盲区占用状态。
- 驾驶员疲劳分数、注意力分数和头部姿态。
- IMU 姿态异常分数。
- 心率异常分数。

### 5.2 风险评分

可以将系统总风险建模为：

$$
R = w_1 R_{front} + w_2 R_{rear} + w_3 R_{driver} + w_4 R_{imu} + w_5 R_{hr}
$$

其中：

- $R_{front}$ 表示前向碰撞风险。
- $R_{rear}$ 表示后向盲区和追尾风险。
- $R_{driver}$ 表示疲劳和分神风险。
- $R_{imu}$ 表示姿态异常风险。
- $R_{hr}$ 表示心率异常风险。

权重可根据比赛演示场景动态调整，例如在拥堵城区提升前向风险权重，在连续长骑场景提升疲劳风险权重。

### 5.3 分级干预策略

系统采用四级分层处置机制：

1. L0 正常：仅显示状态信息，不触发提醒。
2. L1 提示：屏幕提示或短语音提醒，如“注意前方行人”。
3. L2 预警：语音 + 把手振动联合提醒，如“后方有来车，请勿变道”。
4. L3 紧急：在低速、直线、附着条件可接受的前提下触发受控减速，同时发出最高级语音警报。

这里特别强调，两轮电动车不适合照搬汽车的强制自动刹停逻辑。比赛方案中应将“自动刹车”定义为“低速场景下的受控减速辅助”，并加入以下安全约束：

- 仅在极高碰撞概率条件下触发。
- 仅在车速低于设定阈值时允许触发。
- 当 IMU 检测到明显侧倾或弯道姿态时禁止主动制动。
- 驾驶员手动制动优先级高于系统控制。

这一设计更符合两轮车安全常识，也更容易在答辩时自圆其说。

## 6. 软件架构设计

系统软件基于 C++20、ROS2、OpenCV 和 ONNX Runtime 实现，采用模块化 ROS2 节点架构。

### 6.1 ROS2 节点划分

建议划分为以下节点：

1. camera_front_node：采集前向摄像头图像。
2. camera_rear_node：采集后向鱼眼摄像头图像。
3. camera_driver_node：采集驾驶员摄像头图像。
4. front_perception_node：执行目标检测、车道线检测和前向风险分析。
5. rear_perception_node：执行后向来车与盲区分析。
6. driver_monitor_node：执行疲劳、分神和头部姿态分析。
7. imu_node：读取姿态数据并输出平滑结果。
8. mmwave_node：读取雷达测距和心率相关数据。
9. fusion_decision_node：融合多源结果并输出风险等级和控制建议。
10. hmi_node：负责屏幕 UI、语音播报和把手振动控制。
11. brake_interface_node：负责制动联动接口和安全限幅。
12. app_gateway_node：负责蓝牙通信、状态同步和参数配置。

### 6.2 典型话题设计

建议定义以下核心 ROS2 Topic：

- /camera/front/image_raw
- /camera/rear/image_raw
- /camera/driver/image_raw
- /perception/front/objects
- /perception/front/lane
- /perception/rear/targets
- /driver/state
- /imu/state
- /mmwave/state
- /decision/risk_level
- /decision/warning_cmd
- /decision/brake_cmd
- /hmi/display_state
- /app/sync_state

### 6.3 软件处理流程

整体流程如下：

1. 多路传感器采集原始数据。
2. 各感知节点完成独立推理与特征提取。
3. 决策节点进行时间同步、风险打分和分级判断。
4. HMI 节点输出语音、振动和屏显。
5. 满足受控减速条件时，由 brake_interface_node 执行有限干预。
6. app_gateway_node 将关键信息同步到手机 App。

## 7. 模型选型与部署方案

### 7.1 模型选型

1. 目标检测模型：使用 YOLOv26n 进行前向目标检测，兼顾检测精度和边缘端实时性。
2. 路面风险检测模型：针对水坑、坑洼、减速带和障碍物，可在目标检测模型类别中扩展路障类别，或增加轻量级路面缺陷检测分支。
3. 车道线检测模型：使用 LaneATT 进行车道线识别，输出车道线位置和偏移趋势。
4. 驾驶员状态监测模型：采用轻量 CNN 或人脸关键点模型，实现闭眼、打哈欠、头部姿态和分神识别。

如果后期测试发现 YOLOv26n 在 RK3588 上推理压力较大，可准备一套轻量备选模型用于答辩时对比展示，例如更小输入分辨率版本或蒸馏模型版本。

### 7.2 边缘部署方式

基础部署路线如下：

- 开发语言：C++20。
- 通信框架：ROS2。
- 视觉处理：OpenCV。
- 推理框架：ONNX Runtime。
- 构建系统：CMake。
- 目标平台：RK3588，Arm64 交叉编译。

建议在文档中明确区分两个阶段：

1. 基线版本：所有模型通过 ONNX Runtime 在 CPU 或 GPU 上运行，优先保证流程跑通。
2. 优化版本：对主检测模型进一步量化或迁移到 RK3588 NPU 方案，以提升帧率和功耗表现。

这样写的好处是工程路线更务实，不会在早期被底层加速适配卡住。

### 7.3 实时性目标

建议设置如下比赛指标：

- 前向感知帧率：大于等于 20 FPS。
- 后向感知帧率：大于等于 15 FPS。
- 驾驶员监控帧率：大于等于 15 FPS。
- 决策链路端到端延迟：小于 150 ms。
- 风险预警触发延迟：小于 200 ms。
- 平台整机平均功耗：根据散热条件控制在可演示范围内。

## 8. 硬件部署与安装方案

### 8.1 传感器安装建议

- 前向摄像机安装在车头中轴附近，尽量减少偏角误差。
- 后向鱼眼摄像机安装在车尾中高位置，保证左右盲区覆盖。
- 驾驶员摄像机安装在仪表区域上方或车把中部，稳定朝向驾驶员面部。
- IMU 安装在车体刚性较高位置，减少结构件松动带来的噪声。
- 毫米波雷达需兼顾前向测距与心率检测需求，若两种需求难以由同一器件同时满足，建议比赛方案中拆分为“前向测距雷达”和“驾驶员生命体征雷达”两种逻辑角色进行说明。

### 8.2 计算与通信

- 车载主控：RK3588。
- 本地存储：用于缓存模型、日志和短时事件录像。
- 蓝牙模块：与手机 App 通信。
- 音频模块：连接扬声器实现语音告警。
- 驱动接口：连接振动电机和制动执行机构。

## 9. 人机交互与手机 App 设计

### 9.1 车机界面

车机屏幕建议展示以下内容：

1. 前后方实时画面与检测结果叠加。
2. 当前风险等级和告警原因。
3. 驾驶员疲劳状态、注意力状态和心率区间。
4. 当前姿态角、异常动作提示和系统运行状态。

### 9.2 手机 App 功能

手机 App 通过蓝牙与车载系统连接，可提供以下能力：

1. 实时查看当前驾驶状态和设备连接状态。
2. 查看历史告警记录、疲劳统计和高风险事件。
3. 配置预警灵敏度、振动强度、语音音量和受控减速阈值。
4. 导出比赛演示数据和运行日志。

## 10. 比赛演示方案

为了增强项目展示效果，建议设计一套可重复演示的闭环场景：

1. 前方突然出现行人模型，系统触发语音预警。
2. 后方有来车接近，同时模拟驾驶员打转向意图，系统触发盲区预警。
3. 驾驶员连续闭眼或低头，系统触发振动和疲劳提醒。
4. 模拟低速紧急接近障碍物，系统触发受控减速辅助。
5. 手机 App 同步展示当前风险等级与事件记录。

答辩时可重点强调“多模态融合不是简单堆传感器，而是通过风险分级实现更稳健、更少误报的边缘智能辅助系统”。

## 13. 风险与对策

1. 光照变化大导致视觉模型性能下降。
对策：加入夜间和逆光样本，设置传统视觉补偿策略。

2. 多传感器时间不同步导致融合误差。
对策：在 ROS2 中统一时间戳并设置缓存对齐机制。

3. 两轮车自动制动存在安全争议。
对策：明确系统定位为辅助驾驶，不做高速强制刹停，只做低速受控减速。

4. RK3588 实机算力不足。
对策：预留轻量模型、量化方案和异构加速路线。

## 14. 总结

本系统围绕两轮电动车安全骑行需求，构建了一套面向嵌入式大赛的多模态 AI 边缘部署方案。系统基于 RK3588 在本地完成视觉感知、驾驶员监控、姿态与生理信号融合，并通过语音、振动、屏显和受控减速实现闭环辅助。方案具有明确的工程实现路径、良好的展示性以及较强的比赛主题契合度。

如果后续继续细化，建议优先补三部分内容：系统框图、ROS2 话题表、风险评分阈值表。这样这份文档就可以进一步演进为答辩稿和立项书。

## 15. 系统创新点

为了增强比赛表达效果，建议将本系统的创新点总结为以下五项：

1. 面向两轮电动车场景的辅助驾驶设计。
与汽车辅助驾驶不同，本系统重点考虑两轮车更容易受到侧倾、急刹和盲区影响的特点，采用“预警优先、有限干预、低速受控减速”的设计思路，更符合真实骑行安全需求。

2. 多模态联合感知与融合决策。
系统同时融合前向视觉、后向视觉、驾驶员视觉、IMU 姿态和毫米波雷达心率信息，不依赖单一传感器，从而提升复杂场景下的风险识别鲁棒性。

3. 不只识别车辆和行人，还识别路面风险。
系统前向感知不仅关注车辆、行人和交通标志，还特别检测水坑、减速带、坑洼和道路落物。这一点非常贴合两轮电动车场景，因为这类路面目标往往比普通车辆目标更容易直接引发失稳和摔倒风险。

4. 从感知到执行形成闭环辅助。
系统不是单纯做视觉识别，而是把识别结果直接联动到语音播报、把手振动、车机显示和受控减速，实现从检测到干预的完整闭环。

5. 面向边缘端的可落地部署方案。
系统基于 RK3588、ROS2、OpenCV 和 ONNX Runtime 构建，具备明确的软件架构、部署路径和优化路线，既满足比赛“边缘部署”主题，也具备后续产品化延展空间。

本附录用于辅助绘制完整系统框架流程图。建议采用自左向右、分层展示的方式，将整套系统拆为“传感器层、感知计算层、融合决策层、执行交互层、外部连接层”五层。

### A.1 分层结构

#### 第一层：传感器输入层

该层放在图最左侧，包含以下输入模块：

1. 前向摄像机。
2. 后向鱼眼摄像机。
3. 驾驶员摄像机。
4. IMU/陀螺仪。
5. 毫米波雷达。

其中前三个模块可归类到“视觉输入”，后两个模块归类到“非视觉输入”。

#### 第二层：感知计算层

该层放在图中左部，直接接收传感器数据，建议拆成四个并列功能框：

1. 前向环境感知模块。
内容包括：目标检测、车道线检测、TTC 估计、前向风险输出。

2. 后向盲区感知模块。
内容包括：后车检测、侧后方目标检测、盲区占用分析、超车风险输出。

3. 驾驶员监控模块。
内容包括：闭眼检测、打哈欠检测、低头检测、头部姿态与注意力分析。

4. 姿态与生理监测模块。
内容包括：侧倾检测、急转急刹识别、颠簸识别、心率异常识别。

#### 第三层：融合决策层

该层放在图中央，是整个系统核心，建议用一个较大的核心框表示“多模态融合与决策中心”，内部可细分三个子模块：

1. 时间同步与数据对齐。
2. 风险评分与权重融合。
3. 分级告警与受控减速决策。

在这个核心框旁边可以标注风险等级输出：L0、L1、L2、L3。

#### 第四层：执行与交互层

该层放在图右部，接收融合决策层输出，建议拆成四个执行模块：

1. 语音播报模块。
2. 把手振动模块。
3. 车机显示模块。
4. 制动联动模块。

其中制动联动模块旁边建议额外标注一句“小字说明”：仅低速受控减速，不做高速强制刹停。

#### 第五层：外部连接层

该层可放在图最右侧或图下方，表示系统扩展接口：

1. 手机 App。
2. 蓝牙通信模块。
3. 数据记录与日志模块。

App 与车机系统之间通过蓝牙双向连接，可展示“状态同步”和“参数配置”两条箭头。

### A.2 建议箭头关系

绘图时建议明确以下数据流向：

1. 前向摄像机指向前向环境感知模块。
2. 后向鱼眼摄像机指向后向盲区感知模块。
3. 驾驶员摄像机指向驾驶员监控模块。
4. IMU/陀螺仪和毫米波雷达共同指向姿态与生理监测模块。
5. 四个感知模块共同指向多模态融合与决策中心。
6. 多模态融合与决策中心分别指向语音播报、把手振动、车机显示和制动联动。
7. 多模态融合与决策中心同时指向数据记录与日志模块。
8. 数据记录与日志模块与手机 App 之间建立同步箭头。
9. 手机 App 通过蓝牙通信模块反向指向系统参数配置入口。
